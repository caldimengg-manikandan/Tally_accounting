# Financial System Architecture — Tally Replica

This document outlines the technical architecture, security model, and data flow of the Tally Replica cloud accounting platform.

---

## 🏗️ System Overview

The system is designed as a **Multi-Tenant SaaS** (Software as a Service) application, ensuring that multiple companies can use the same infrastructure while maintaining strict data isolation.

### 🔌 Backend (Node.js & Express)
- **Modular Design:** Each business domain (Sales, Inventory, Accounting) is isolated into its own module with dedicated controllers and routes.
- **Unified Ledger Engine:** Powered by `AccountingService.js`, which ensures all "Double-Entry" transactions are balanced and atomic.
- **ORM (Sequelize):** Provides a database-agnostic layer supporting PostgreSQL, MySQL, and SQLite.

### ⚛️ Frontend (React & Vite)
- **State Management:** Uses **Redux Toolkit** for complex financial state and **Zustand** for lightweight UI persistence.
- **Data Rendering:** **AG-Grid** handles enterprise-grade sorting, filtering, and cell editing for high-volume ledger data.
- **Animations:** **Framer Motion** ensures a premium, fluid user experience during navigation and data entry.

---

## 🔐 Security & Multi-Tenancy

### 🏘️ Tenant Isolation
- **`CompanyId` Scoping:** Every record in the system (Ledgers, Items, Vouchers, Transactions) is linked to a `CompanyId`.
- **Tenant Middleware:** The `tenantAccess` guard forces all queries to include `where: { CompanyId: req.user.activeCompanyId }`.
- **Cross-Tenant Prevention:** Users can only act within their "active" company context, preventing accidental data leakage.

### 👮 Role-Based Access Control (RBAC)
The system implements a granular role matrix:

| Role | Access Level | Responsibilities |
|---|---|---|
| **SUPER_ADMIN** | Global | Platform maintenance, cross-company billing, global audit. |
| **ADMIN** | Company Owner | User management, company settings, full financial rights. |
| **ACCOUNTANT** | Power User | Create/Edit Vouchers, reconcile accounts, manage ledgers. |
| **MANAGER** | Reviewer | Approve transactions, view reports, supervise accountants. |
| **AUDITOR** | Specialized Read | Audit log visibility, read-only financial transparency. |
| **VIEWER** | Basic Read | View dashboard and general reports, no editing power. |

## 📈 Core Workflows

### 1. The Journal Entry Flow
1. **Frontend Request:** User submits a Voucher (e.g., Sales Invoice).
2. **Double-Entry Validation:** The `AccountingService` ensures Total Debits = Total Credits.
3. **Atomic Commit:** Uses **Sequelize Transactions** to ensure a Voucher is only created if all corresponding entries succeed.
4. **Audit Logging:** The system records the action, timestamp, and the user who originated the entry.

### 2. Live Reporting Flow
1. **Dynamic Totals:** Reports like **Trial Balance** and **Balance Sheet** are generated by aggregating transaction rows in real-time.
2. **Soft Deletes:** Records are never permanently removed; they are "archived" (using `paranoid: true`), allowing for audit recovery.

## 📝 Compliance & Auditing
To meet accounting standards, the platform includes:
- **Audit Logs:** Tracks every critical modification with original and updated data snapshots (`oldData` / `newData`).
- **Creator Tracking:** Every transaction is tagged with a `createdBy` UUID to enforce accountability.
- **Financial Year Constraints:** Transactions are validated against the company's defined financial year boundaries.
